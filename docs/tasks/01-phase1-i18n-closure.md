# Task 01：Phase 1 `i18n` 完整闭环

> 任务编号：`Task 01`
> 优先级：`P1`
> 当前状态：`已完成`
> 最近更新：`2026-07-06`
> 当前进度：`7/7`

---

## 1. 任务目标

把当前“显式 helper + 局部英文辅助输出”的 `i18n` 骨架，收口成符合 `docs/DESIGN.md` 预期的完整语言层，实现：

- 统一语言包组织；
- 占位符或等价模板化输出策略；
- 核心输出链路可切换语言；
- 避免继续在各模块散落翻译逻辑。

---

## 2. 当前现状

已完成：

- `Language` 枚举已存在；
- `Solar` / `Lunar` / `Foto` / `Tao` / `NineStar` 已有显式语言辅助方法；
- 第一批 typed culture 已增加局部语言方法；
- `cargo test --features i18n` 当前通过。

当前闭环结论：

- 已建立统一语言资源层；
- 核心完整字符串输出已收口到统一模板入口；
- typed/domain 新旧对象已共用同一批语言 helper；
- 对节名/专名采用统一策略：
  - 已知条目翻译；
  - 未收录条目保留原名；
  - 不再由业务模块各自决定是否翻译。

---

## 3. 实现范围

### 3.1 本任务已完成内容

1. 梳理当前全部 `*_in_lang(...)` 的分布和重复逻辑。
2. 设计统一语言资源层：
   - 常量表；
   - 模板表；
   - 或占位符替换表。
3. 给以下对象收口语言输出策略：
   - `Solar`
   - `Lunar`
   - `Foto`
   - `Tao`
   - `NineStar`
   - typed culture 新旧对象
4. 明确“默认中文直出”和“启用 i18n 的语言切换”之间的边界。
5. 让新增 typed 对象不再各自复制翻译表。
6. 补充语言输出测试矩阵。

### 3.2 本任务不包含

- 新增第三种以上语言；
- 改造外部 API 为全局运行时语言切换器；
- 重写所有中文业务文本为自然英文；
- README/网站国际化文案工程。

---

## 4. 建议拆分步骤

### Step 1：盘点与归并

- 列出当前所有 `*_in_lang(...)` 方法；
- 归类出：
  - 基础词表；
  - 组合词表；
  - 完整字符串模板；
  - 临时直拼逻辑。

### Step 2：抽象统一语言资源层

- 建立集中式 `i18n` 资源入口；
- 决定是否采用：
  - 占位符模板重写；
  - 统一 formatter；
  - 或混合方案。

### Step 3：收口完整字符串输出

- 优先收口：
  - `Solar::to_full_string_in_lang`
  - `Lunar::to_full_string_in_lang`
  - `Foto::to_full_string_in_lang`
  - `Tao::to_full_string_in_lang`
- 减少重复拼接与硬编码英文片段。

### Step 4：typed culture 对齐

- 让 typed domain 对象优先走统一语言资源层；
- 避免在 `culture.rs` 中继续增长零散翻译分支。

### Step 5：测试闭环

- 扩展 `tests/i18n.rs`；
- 覆盖：
  - 基础词条；
  - 完整字符串；
  - typed 对象；
  - 节气/生肖/方位/值星/月相等高频输出。

---

## 5. 验收结果

已满足：

1. `i18n` 输出路径已有统一资源层；
2. 核心完整字符串输出已不再散落大量重复英文硬编码；
3. typed culture 新旧对象已使用一致的语言策略；
4. `cargo test --features i18n` 已全绿；
5. 文档已明确默认中文与显式语言输出边界。

---

## 6. 风险点

- 过度追求“完全复刻 JS 占位符重写”可能引入不必要复杂度；
- 如果一次性重构过大，容易影响现有字符串对齐行为；
- 多处 `to_full_string_in_lang` 收口时要避免破坏现有测试口径。
